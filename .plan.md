# 物流查询增强实施方案

## Context

台账管理系统当前物流功能存在三个不足：
1. 记录列表中快递单号只是纯文本，无法点击查看物流详情
2. 物流状态只能手动逐条或批量同步，没有定时自动查询
3. 快递查询页面只显示最近10条物流记录，不支持按记录ID查询

本方案将补齐这三个功能，涉及后端 (FastAPI + APScheduler) 和前端 (单文件 index.html)。

---

## Feature 1: 记录列表 — 点击快递单号弹窗显示物流详情

### 修改文件：`frontend/index.html`

**1.1 新增物流详情弹窗 HTML**（放在 `<!-- ========== Edit Modal -->` 之前，约 line 1465）

```html
<!-- ========== Tracking Detail Modal ========== -->

            </div>
        </div>
    </div>

```

**1.2 快递单号改为可点击链接**（line 1973）

将原来的纯文本改为带链接样式的按钮：
```javascript
// 之前
`<td ...>${r.tracking_no || '-'}</td>`
// 之后
`<td style="font-size:12px;max-width:120px">
    ${r.tracking_no
        ? `<a href="javascript:void(0)" onclick="showTrackingDetail(${r.id})" style="color:var(--primary);text-decoration:underline;cursor:pointer" title="点击查看物流">${r.tracking_no}</a>`
        : '-'}

```

**1.3 新增 `showTrackingDetail(recordId)` JS 函数**（放在 `syncTracking()` 之后）

调用已有的 `GET /api/tracking/status/{record_id}` 接口，在 modal 中显示完整物流 timeline（复用 tracking-timeline 样式）。

## Feature 2: 定时自动查询物流状态

### 修改文件：`backend/requirements.txt`
新增依赖：
```
apscheduler==3.10.4
```

### 修改文件：`backend/database.py`
在 `_init_db()` 中新增 `site_config` 表（用于存储 cron 表达式等配置）：

```python
await db.execute("""
    CREATE TABLE IF NOT EXISTS site_config (
        key TEXT PRIMARY KEY,
        value TEXT NOT NULL DEFAULT ''
    )
""")
await db.execute("""
    INSERT INTO site_config (key, value) VALUES ('tracking_cron', '0 */3 * * *')
    ON CONFLICT (key) DO NOTHING
""")
```

### 新增文件：`backend/services/scheduler.py`

核心逻辑：
- `validate_tracking_no(no)` — 校验快递单号格式（至少6位字母数字）
- `auto_sync_job()` — 查询数据库中 `tracking_state != '3'`（未签收）且有单号的记录，先校验格式，跳过无效单号，再调用 `batch_query_tracking` 更新状态
- `refresh_schedule(cron_expr)` — 解析 cron 表达式并重新设置 APScheduler job
- `start_scheduler()` / `stop_scheduler()` — 应用启动/关闭时调用

### 修改文件：`backend/main.py`

在 lifespan 中集成调度器：
```python
from services.scheduler import start_scheduler, stop_scheduler

@asynccontextmanager
async def lifespan(app):
    await init_pool()
    async for db in get_db():
        await ensure_admin(db)
    await start_scheduler()   # 新增
    yield
    await stop_scheduler()    # 新增
    await close_pool()
```

### 修改文件：`backend/routers/tracking.py`

新增两个接口：
- `GET /api/tracking/cron` — 获取当前 cron 表达式
- `PUT /api/tracking/cron` — 更新 cron 表达式（校验格式、存库、刷新调度器）

### 修改文件：`frontend/index.html`

在快递查询页面（`page-tracking`）底部新增定时任务配置区：
- cron 输入框 + 保存按钮
- 格式提示（分 时 日 月 周）

对应 JS：`loadCronSetting()` 和 `saveCron()` 函数。在 `switchPage('tracking')` 时加载当前设置。

## Feature 3: 快递查询页面支持按记录ID查询 + 全量显示历史

### 修改文件：`backend/services/kuaidi100.py`

将 `query_tracking` 中 history 的 `[:10]` 限制去掉，全量返回：

```python
# 之前
"history": [... for item in (data.get("data") or [])[:10]],
# 之后 —— 全量返回
"history": [... for item in (data.get("data") or [])],
```

### 修改文件：`frontend/index.html`

在快递查询页面新增"按记录ID查询"输入框和按钮：
- 新增 `queryTrackingByRecord()` 函数，调用 `/api/tracking/status/{record_id}`
- 抽取公共渲染函数 `renderTrackingResult(d, container)` 供 `queryTracking` 和 `queryTrackingByRecord` 共用

---

## 实施顺序

1. **Feature 1**（前端改动为主，无新依赖）— 最小改动，最快可见效果
2. **Feature 3**（前端 + 后端小改动）— 去掉 history 限制 + 按记录ID查询
3. **Feature 2**（新增依赖 + 新文件 + 改 main.py）— 最复杂，最后实施

## 验证方法

1. **Feature 1**: 在记录列表点击一条有快递单号的记录，应弹窗显示完整物流 timeline
2. **Feature 2**: 
   - 启动后端，查看日志应显示 `[调度器] 已设置定时物流查询: 0 */3 * * *`
   - 访问快递查询页面，修改 cron 表达式并保存，确认调度已更新
3. **Feature 3**: 
   - 在快递查询页面输入记录ID，点击"按记录查询"，应显示该记录的完整物流历史（不受10条限制）
   - 手动查询一个快递单号，确认显示全量历史记录
